### 2. Assessment  
**Critically Vulnerable**  

#### Why?  
- **Service Account Misconfiguration**: `sql_svc` is a user account (not a computer account), which is a critical error. Service accounts for SQL Server should be **computer accounts** (e.g., `SQLServer$<domain>`) to enforce proper service credential handling and minimize attack surface.  
- **Password Exposure Risk**: While passwords are not stored in plaintext in AD objects, the description field for `samwell.tarly` explicitly states `(Password : Heartsbane)`, indicating improper documentation of passwords in the AD schema. This is a severe violation of security best practices (passwords should never be visible in AD objects).  
- **Privilege Escalation Path**: User accounts like `sql_svc` (if misconfigured) can be targeted for attacks like Kerberoasting (see Attack Path section).  

---

### 3. Attack Path  
**Kerberoasting**  
- **Technical Impact**: Kerberoasting exploits the Kerberos protocol to request service tickets (TGS tickets) for user accounts with service identities (e.g., `sql_svc`). If the service account’s password is weak or guessable (e.g., `Heartsbane`), attackers can crack the password hashes offline to gain access to the SQL Server or escalate privileges.  
- **Why It Works**:  
  - `sql_svc` is a user account (not a computer account), making it eligible for Kerberoasting.  
  - The description’s mention of `(Password : Heartsbane)` suggests a weak password, which is easily brute-forced or cracked.  
  - If `sql_svc` has elevated permissions (e.g., access to sensitive data), attackers can exploit this to compromise the entire domain.  

---

### 4. Remediation  
**Step-by-Step Fix**:  

#### Step 1: Convert `sql_svc` to a Computer Account  
1. **Create a Valid Computer Account**:  
   - In Active Directory Users and Computers, right-click **Computers** → *New → Computer* → Name it `SQLServer$northsevenkingdoms.local` (or similar).  
   - This ensures the account is a computer account (not a user account), which is required for service accounts.  
2. **Configure SQL Server to Use the Computer Account**:  
   - Open SQL Server Configuration Manager → Navigate to **SQL Server Services** → Right-click the SQL Server service → *Properties* → *Log On* tab.  
   - Set the **Service account** to `SQLServer$northsevenkingdoms.local` (or the newly created computer account).  
   - Restart the SQL Server service.  

#### Step 2: Remove the Misconfigured User Account  
1. **Delete `sql_svc`**:  
   - Right-click `sql_svc` in the Users container → *Delete*.  
   - Verify that all SQL Server services are configured to use the new computer account.  

#### Step 3: Address Password Documentation Risk  
1. **Never Store Passwords in Description Fields**:  
   - Edit the description of `samwell.tarly` (and other accounts) to remove any reference to passwords (e.g., change `"(Password : Heartsbane)"` to `"(Account Purpose: SQL Service)"`).  
   - Ensure all administrators are trained to avoid documenting passwords in AD objects.  
2. **Enforce Strong Password Policies**:  
   - Use Group Policy (`Computer Configuration → Windows Settings → Security Settings → Account Policies → Password Policy`) to enforce:  
     - Minimum password length (≥12 characters).  
     - Password complexity (uppercase, lowercase, numbers, symbols).  
     - Password expiration (e.g., 90 days).  

#### Step 4: Verify Group Memberships  
1. **Check `Domain Admins` Membership**:  
   - Ensure no unnecessary user accounts are members of `Domain Admins`. If `sql_svc` was previously in `Domain Admins`, remove it immediately.  
2. **Audit Service Account Permissions**:  
   - Use **Active Directory Users and Computers** → Right-click `sql_svc` → *Properties* → *Security* tab → Verify that permissions are limited to the SQL Server service.
