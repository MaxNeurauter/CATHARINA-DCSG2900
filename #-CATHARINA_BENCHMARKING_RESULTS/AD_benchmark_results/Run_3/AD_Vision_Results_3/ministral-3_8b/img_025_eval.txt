  - Could lead to **pass-the-hash** or **pass-the-ticket** attacks if combined with other misconfigurations (e.g., unconstrained delegation).

#### **B. Privilege Escalation (Indirect)**

#### **C. Denial-of-Service (DoS) via Password Lockouts**

### **4. Remediation: Step-by-Step Fix**
#### **Goal:** Restrict password change permissions to only necessary groups and enforce least privilege.

#### **Steps:**
1. **Open Active Directory Users and Computers**
   - Navigate to the user (`robb.stark`) and right-click → **Properties**.

2. **Go to the Security Tab**
   - Click **Advanced** to access the **Advanced Security Settings**.

3. **Remove "Change Password" Permission for Everyone**
   - Under **Permission Entries**, select the entry for **Everyone** → **Edit**.
   - Remove **"Change password"** from the **Allowed** list.
   - Click **OK** to save changes.

4. **Restrict Password Change to Specific Groups (Recommended)**
   - Add **Domain Admins** (or a dedicated password management group) with **"Change password"** permission.
   - Remove **"Everyone"** entirely if not required for other purposes.

5. **Audit Changes**
   - Verify the permissions by checking the **Effective Access** tab.
   - Ensure no unintended permissions remain.

6. **Monitor for Unauthorized Access**
   - Use **Security Event ID 4724** (Directory Service Access) in **Windows Event Logs** to detect suspicious password changes.

7. **Enforce Strong Password Policies (Optional but Recommended)**
   - Set a **Password Never Expires** flag if the account is a service account.
   - For regular user accounts, enforce **complex password requirements** via **Group Policy**.

8. **Document Changes**
   - Record the changes in your **change management system** for compliance.

### **Additional Best Practices**
- **Disable Inheritance** (if not already done) to avoid unintended permission propagation.

- **Regularly Audit Permissions** using **PowerShell** (`Get-ADUser` + `AccessControl`) or **Group Policy Preferences**.
