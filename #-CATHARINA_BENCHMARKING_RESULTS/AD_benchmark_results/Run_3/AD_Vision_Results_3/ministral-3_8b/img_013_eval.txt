### **3. Attack Paths & Technical Impact**
#### **A. AS-REP Roasting (Kerberos Attack)**
- **Vector:** `Authenticated Users` have **"Enable per-user reversibly encrypted password"** (AS-REP pre-authentication disabled).
- **Impact:**
  - Attackers can **roast Kerberos tickets** for users with **no pre-auth required**, cracking hashed passwords offline.
  - If **Service Principal Names (SPNs)** are exposed, attackers can **target privileged accounts** (e.g., service accounts).
- **Detection:** Use **BloodHound** or **ADAuditPlus** to check for users with **no pre-auth required**.

#### **B. Information Disclosure (Anonymous Access)**
- **Vector:** `ANONYMOUS LOGON` has **Read access to descendant objects**.
- **Impact:**
  - Attackers can **enumerate users, groups, and OUs** via **LDAP anonymous queries**.
  - Enables **reconnaissance** for further attacks (e.g., **Golden Ticket, Pass-the-Hash**).
- **Detection:** Check **LDAP anonymous bind** permissions in **ADAC (Active Directory Administrative Center)**.

#### **C. Privilege Escalation (SELF Write Permissions)**
- **Vector:** `SELF` has **"Validated write to computer attributes"** on descendant objects.
- **Impact:**
  - If an attacker gains **local admin on a machine**, they could **modify computer attributes** (e.g., **msDS-AllowedToActOnBehalfOfOtherIdentity**).
  - Enables **impersonation of other users** (e.g., **via "RunAs"**).
- **Detection:** Audit **computer object modifications** in **Security Event Log (Event ID 4663)**.

---
### **4. Remediation: Step-by-Step Fixes**
#### **A. Mitigate AS-REP Roasting**
1. **Disable "Enable per-user reversibly encrypted password" for Authenticated Users**:
   - Go to **Active Directory Users and Computers** → **Properties of the OU/GPO** → **Security** → **Edit**.
   - Remove the **ACE** granting `Authenticated Users` **"Enable per-user reversibly encrypted password"**.
   - **Replace with:**
     - `Authenticated Users` → **Deny** → **Read password reset info** (if possible).
     - **Enforce "Users must change password at next logon"** via **Default Domain Policy (GPO)**.

2. **Enforce "No pre-authentication required" only for specific accounts**:
   - Use **Group Policy Preferences (GPP)** to **disable "No pre-auth"** for all users except **service accounts**.
   - **Check SPNs** with:
     ```powershell
     Get-ADUser -Filter * -Properties ServicePrincipalName | Where-Object {$_.ServicePrincipalName -ne $null}
     ```

#### **B. Restrict Anonymous Access**
1. **Remove Anonymous Read Access**:
   - In **Advanced Security Settings**, **remove the `ANONYMOUS LOGON` "Read" permission** from descendant objects.
   - **Replace with:**
     - `ANONYMOUS LOGON` → **Deny** → **Read** (if possible).
   - **Alternative:** Use **LDAP Signing/Signing + Encryption** in **Domain Controller Security Policies**.

2. **Audit Anonymous Bind Attempts**:
   - Enable **Event ID 5136** (LDAP anonymous bind attempts) in **Security Log**.

#### **C. Harden SELF Write Permissions**
1. **Restrict SELF to Only Its Own Object**:
   - Modify `SELF` permissions to **only apply to "This object only"** (not descendants).
   - Remove **"Validated write to computer attributes"** from descendant objects.

2. **Audit Computer Attribute Modifications**:
   - Enable **Advanced Audit Policy Configuration** → **Audit Object Access** (Event ID 4663).

#### **D. General Hardening**
1. **Remove Legacy Permissions**:
   - Remove all **Pre-Windows 2000 Compatible Access** entries unless absolutely necessary.

2. **Apply Least Privilege to Domain Admins**:
   - **Remove `Domain Admins` from `Enterprise Admins`** (segregation of duties).
   - **Use Just Enough Administration (JEA)** for admin tasks.

3. **Enable GPO Auditing**:
   - **Computer Configuration → Policies → Windows Settings → Security Settings → Advanced Audit Policy Configuration**
   - Enable:
     - **Audit Directory Service Changes** (Event ID 5136, 5137, 5138).
     - **Audit Account Management** (Event ID 4720, 4722).
