  - **Microsoft network server: Digitally sign communications (always)** → **Enabled**
  - **Microsoft network server: Digitally sign communications (if client agrees)** → **Enabled**

---

### **2. Assessment**
The configuration is **Misconfigured** with a **Critical Vulnerability**.

#### **Specific Issue:**
- **LDAP Server Signing Requirement (Domain Controller) is set to "None"**
  - This means **LDAP traffic (port 389, unencrypted) is not signed or encrypted**, exposing credentials and sensitive data to **Man-in-the-Middle (MITM) attacks, eavesdropping, and replay attacks**.

- While **secure channel signing (Kerberos) and Microsoft network signing** are enabled, **LDAP remains unprotected**, which is a **major weakness** in Active Directory security.

---

### **3. Attack Path & Technical Impact**
#### **Attack Vector: LDAP Relay/Man-in-the-Middle (MITM) & Credential Harvesting**
- **LDAP (Lightweight Directory Access Protocol) is transmitted in plaintext** when signing is disabled.
- An attacker can:
  - **Eavesdrop** on LDAP traffic to capture usernames, passwords, and sensitive attributes.
  - **Perform LDAP Relay Attacks** (e.g., using tools like **Responder** or **Evilginx**) to intercept and replay authentication requests.
  - **Exfiltrate Active Directory data** (user lists, group memberships, computer accounts).
- **Impact:**
  - **Credential Theft** (passwords, hashes, Kerberos tickets).
  - **Privilege Escalation** (if an attacker gains a low-privilege account, they can move laterally).
  - **Account Takeover** (via replay attacks on LDAP-bound protocols like RDP, SMB, or Exchange).

#### **Why Not Other Attacks?**
- **Kerberoasting/AS-REP Roasting:** Not directly applicable here (requires SPN enumeration or pre-auth failures).
- **Silver Ticket:** Requires Kerberos delegation misconfigurations (not present here).
- **Golden Ticket:** Requires compromised **krbtgt** hash (not directly tied to LDAP signing).

---

### **4. Remediation (Step-by-Step Fix)**
#### **Immediate Fix (Critical):**
1. **Enable LDAP Signing on Domain Controllers**
   - Open **Group Policy Management Console (GPMC)**.
   - Navigate to:
     **`Default Domain Controller Policy` → Computer Configuration → Policies → Windows Settings → Security Settings → System Settings → LDAP Client Signing Requirements`**
   - Set **"LDAP server signing requirements"** to:
     - **`Require signing`** (for maximum security).
     - **At minimum, set to `Require signing`** (not "Never" or "Anonymous").

2. **Apply the Policy to Domain Controllers**
   - Run `gpupdate /force` on all Domain Controllers.
   - Verify with:
     ```powershell
     Get-ItemProperty -Path 'HKLM:\SYSTEM\CurrentControlSet\Control\Lsa' | Select LDAPServerIntegrity
     ```
     (Should return `2` for **LDAP_SIGN_REQ**)

3. **Force a Replication & Restart (if needed)**
   - If changes don’t apply immediately, restart the **Active Directory Domain Services** service or reboot DCs.

#### **Additional Hardening (Recommended):**
- **Enable LDAPS (LDAP over TLS/SSL)**
  - Configure **TLS 1.2+** for LDAP (port 636) to encrypt traffic.
  - Issue a **Certificate Authority (CA)-signed certificate** for LDAP.
- **Restrict LDAP Access via Firewall**
  - Allow LDAP (389/636) **only** from trusted IPs (e.g., internal network).
- **Audit LDAP Access**
  - Enable **Advanced Audit Policy Configuration** for **Directory Service Access** events (Event ID 4662).

#### **Verification:**
- Test with:
  ```powershell
  Test-LdapConnection -Server <DC_Name> -Verbose
  ```
  (Should return **`SigningRequired`** or **`SigningOptional`** but not **`SigningNotRequired`**).

---
### **Final Note:**
This is a **high-severity misconfiguration** that should be remediated **immediately**, especially in environments handling sensitive data. Delaying this fix exposes the domain to **credential theft and lateral movement attacks**.
