### **3. Attack Paths and Technical Impact**
#### **A. Kerberoasting (Exploitable)**
- **Vector**: If Domain Admins or service accounts have **passwords set to "never expire"** and are **not in the "Protected Users" group**, an attacker can:
  1. Request SPNs (Service Principal Names) for these accounts.
  2. Crack hashed passwords offline (Kerberoasting).
  3. Gain Domain Admin privileges.
- **Impact**: Full domain compromise.

#### **B. AS-REP Roasting (Exploitable)**
- **Vector**: If accounts (e.g., `Administrator`, `brandon.stark`) have **"Do not store password"** or **"Password never expires"** enabled, an attacker can:
  1. Request AS-REP (Authentication Server Response) for these accounts.
  2. Crack the hash offline.
  3. Gain control of the account.
- **Impact**: Privilege escalation to Domain Admin.

#### **C. Privilege Escalation (Exploitable)**
- **Vector**: Fictional usernames (`jon.snow`) suggest **weak password policies** (e.g., no complexity, no lockout). An attacker could:
  1. Guess or brute-force passwords for these accounts.
  2. Add them to **Domain Admins** or **Enterprise Admins**.
  3. Take over the domain.
- **Impact**: Full domain compromise.

#### **D. Social Engineering (Exploitable)**
- **Vector**: Usernames like `hodor` or `rickon.stark` are **easy to guess** and could be used in:
  1. Phishing emails (e.g., "Your password for `rickon.stark` has expired").
  2. Credential harvesting via fake login pages.
- **Impact**: Initial access for further exploitation.

#### **E. Golden Ticket Attack (Mitigated but Risky)**
- **Vector**: If `krbtgt` password is compromised (e.g., via Kerberoasting), an attacker can:
  1. Create a **Golden Ticket** to impersonate any user.
  2. Bypass MFA and gain full domain control.
- **Impact**: Full domain takeover.

### **4. Remediation Steps**
#### **Step 1: Restrict Domain Admin Membership**
- **Action**: Remove all non-essential accounts (e.g., `brandon.stark`, `catelyn.stark`) from **Domain Admins**.
- **How**:
  1. Open **Active Directory Users and Computers**.
  2. Navigate to **Domain Admins** group.
  3. Remove unnecessary members.
- **Justification**: Limit blast radius of privilege escalation.

#### **Step 2: Enforce Strong Password Policies**
- **Action**: Set **password complexity requirements** and **expiration** for all accounts.
- **How**:
  1. Open **Group Policy Management**.
  2. Navigate to **Default Domain Policy** > **Computer Configuration** > **Policies** > **Windows Settings** > **Security Settings** > **Account Policies** > **Password Policy**.
  3. Set:
     - **Minimum password length**: 12+ characters.
     - **Password must meet complexity requirements**: Enabled.
     - **Maximum password age**: 90 days.
     - **Minimum password age**: 1 day.
     - **Password never expires**: **Disabled** for all accounts except service accounts (if needed).
- **Justification**: Prevent weak passwords and password reuse.

#### **Step 3: Disable or Restrict Fictional/Unused Accounts**
- **Action**: Disable or rename accounts like `jon.snow`, `arya.stark`, `hodor`, etc.
- **How**:
  1. Right-click each account > **Properties**.
  2. Disable the account or rename it to a **non-descriptive name** (e.g., `user_1234`).
- **Justification**: Remove low-hanging fruit for attackers.

#### **Step 4: Enable "Protected Users" Group for Domain Admins**
- **Action**: Move **Domain Admins** into the **Protected Users** group.
- **How**:
  1. Open **Active Directory Users and Computers**.
  2. Find the **Protected Users** group.
  3. Add **Domain Admins** as a member.
- **Justification**: Prevent Kerberoasting and AS-REP Roasting.

#### **Step 5: Disable the Guest Account**
- **Action**: Disable the built-in **Guest** account.
- **How**:
  1. Open **Computer Management** > **Local Users and Groups**.
  2. Disable the **Guest** account.
- **Justification**: Remove default enabled accounts that can be exploited.

#### **Step 6: Restrict Service Accounts**
- **Action**: Ensure service accounts (e.g., SQL, Exchange) are **not used for human logins** and have **complex passwords**.
- **How**:
  1. Create dedicated service accounts (e.g., `svc_sqlserver`).
  2. Assign **minimum required permissions**.
  3. Set **password complexity** and **expiration**.
- **Justification**: Prevent credential reuse and privilege escalation.

#### **Step 7: Enforce Least Privilege**
- **Action**: Review all groups and ensure **least privilege** is applied.
- **How**:
  1. Audit group memberships (e.g., **Enterprise Admins**, **Schema Admins**).
  2. Remove unnecessary members.
- **Justification**: Limit lateral movement for attackers.

#### **Step 8: Monitor for Suspicious Activity**
- **Action**: Enable **Advanced Auditing** for:
  - Domain Admin logins.
  - Kerberos ticket requests (AS-REP).
  - Privilege escalation attempts.
- **How**:
  1. Open **Group Policy Management**.
  2. Configure **Security Audit Policy** > **Advanced Security Audit Policy Configuration**.
  3. Enable:
     - **Audit Logon Events** (Success/Failure).
     - **Audit Directory Service Changes**.
     - **Audit Account Management**.
- **Justification**: Detect and respond to attacks early.

#### **Step 9: Educate Users on Password Hygiene**
- **Action**: Train users on:
  - Avoiding weak passwords (e.g., `password123`).
  - Not sharing credentials.
  - Recognizing phishing attempts.
- **Justification**: Reduce human error as an attack vector.

#### **Step 10: Regular Audits and Compliance Checks**
- **Action**: Schedule **quarterly audits** of:
  - Password policies.
  - Group memberships.
  - Account lockout thresholds.
- **How**:
  - Use **Microsoft Security Compliance Toolkit** or **Third-party tools** (e.g., CrowdStrike, SentinelOne).
- **Justification**: Maintain ongoing security posture.
