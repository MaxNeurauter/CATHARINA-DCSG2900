### 3. Attack Path  
- **Attack Vector**: *Credential Theft + Persistent Access*.  
- **Technical Impact**: If the account is compromised (e.g., via phishing, password spraying, or insider threat), the absence of an expiration date allows the attacker to retain long-term access. This enables:  
  - Lateral movement across the network.  
  - Data exfiltration or modification.  
  - Privilege escalation (e.g., if the account has group memberships for critical systems).  
  The "Never" expiration eliminates a key control to mitigate the impact of a breach, as a user account (not a service account), the "Account is sensitive" setting is less critical here.  

### 4. Remediation  
1. **Set Account Expiration**:  
   - Select "End of" in the "Account expires" section and assign a reasonable expiration date (e.g., 1 year).  
   - *For all users*: Implement a Group Policy to enforce account expiration (e.g., "User must change password at next logon" + "Account expiration" for non-service accounts).  
2. **Review Privileged Accounts**:  
   - If this account has group memberships for critical systems (e.g., Domain Admins), ensure "Account is sensitive" is *checked* to prevent delegation abuse.  
3. **Audit and Policy Enforcement**:  
   - Run a quarterly audit of all user accounts to verify expiration dates are set.  
   - Use Active Directory tools (e.g., `Get-ADUser -Filter * -Properties AccountExpirationDate`) to automate checks.  

This addresses the immediate risk of persistent access and aligns with Microsoft’s best practices for account lifecycle management.
