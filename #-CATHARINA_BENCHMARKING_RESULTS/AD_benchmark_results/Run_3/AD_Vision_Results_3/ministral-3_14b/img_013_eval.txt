5. **Pre-Windows 2000 Compatible Access entries** → Legacy permissions that may allow weak authentication methods.
6. **Creator Owner has write access on descendant objects** → Could allow privilege escalation if an attacker controls an object’s creation.
7. **Domain/Enterprise Admins have overly permissive access** → While expected, the scope (applying to descendant objects) increases attack surface.

### **4. Remediation**
#### **Step-by-Step Fixes:**

#### **A. Remove ANONYMOUS LOGON Access**
1. Open **Active Directory Users and Computers (ADUC)**.
2. Navigate to the object (**north** OU/Group/Computer).
3. Right-click → **Properties** → **Security** → **Advanced**.
4. Remove the **ANONYMOUS LOGON** entry under **Permission entries**.
5. Click **Apply** → **OK**.

#### **B. Restrict Authenticated Users Permissions**
1. In the same **Advanced Security Settings**, locate **Authenticated Users**.
2. Remove the following permissions:
   - **Enable per user reversibly encrypted password**
   - **Update password not required bit**
   - Any **Special** permissions not explicitly required.
3. Replace with least-privilege access (e.g., **Read** if needed).
4. Click **Apply** → **OK**.

#### **C. Audit and Remove Pre-Windows 2000 Compatible Access**
1. Identify all **Pre-Windows 2000 Compatible Access** entries.
2. Remove them unless absolutely necessary (legacy systems may require them).
3. If unsure, consult Microsoft documentation or a security audit tool (e.g., **BloodHound**).

#### **D. Restrict ENTERPRISE DOMAIN CONTROLLERS Permissions**
1. Locate **ENTERPRISE DOMAIN CONTROLLERS** entries.
2. Remove unnecessary permissions such as:
   - **Replicating Directory Changes**
   - **Replication Synchronization**
   - **Manage replication topology**
3. Keep only **required replication permissions** (e.g., **Replicating Directory Changes All** may still be needed for DCs).

#### **E. Secure Creator Owner and SELF Permissions**
1. Remove **Creator Owner** and **SELF** from having **write access** on descendant objects unless explicitly required.
2. Replace with **inherited permissions** where possible.

#### **F. Enable Kerberos Pre-Authentication (If Not Already Enforced)**
1. Open **Active Directory Users and Computers**.
2. Navigate to **Domain Controllers OU**.
3. For each **Domain Controller**, ensure **"Do not require Kerberos pre-authentication"** is **disabled** (default secure setting).
4. Run:
   ```powershell
   Set-ADAccountControl -Identity "DC_Name" -DoesNotRequirePreAuth $false
   ```

#### **G. Harden Domain/Enterprise Admins**
1. Ensure **Domain Admins** and **Enterprise Admins** do not have **Full Control** over general OUs unless absolutely necessary.
2. Use **group-managed service accounts (gMSA)** for service principals instead of human-managed accounts.
