### **3. Attack Path (If Vulnerable)**
The **only misconfigured setting** is:
- **Force logoff when logon hours expire (Disabled)**

#### **Potential Attack Vector:**
- **Privilege Escalation / Lateral Movement**
  - If an attacker gains initial access (e.g., via phishing, weak credentials, or misconfigured RDP), they could exploit **logon hour restrictions** to maintain persistence.
  - Since logoff is not enforced, an attacker could:
    - Use a compromised account after its allowed logon hours.
    - Perform actions outside authorized time windows (e.g., exfiltrating data during off-hours).
    - Bypass time-based access controls if combined with other misconfigurations (e.g., stale credentials).

#### **Technical Impact:**
- **Increased dwell time** (time an attacker remains undetected).
- **Bypassing time-based security controls** (e.g., if logon hours are set to restrict admin access during non-business hours).
- **Facilitating credential abuse** (e.g., Pass-the-Hash attacks if combined with other weaknesses).

### **4. Remediation**
#### **Step-by-Step Fix for Misconfigured Setting:**
1. **Open Local Security Policy:**
   - Press `Win + R`, type `secpol.msc`, and press **Enter**.

2. **Navigate to Network Security Settings:**
   - Go to:
     **Local Policies** → **Security Options**
   - Look for:
     **Microsoft Network Server: Digitally sign communications (always)**
     *(Note: The exact path for "Force logoff when logon hours expire" may require Group Policy Editor (`gpedit.msc`) if this is a domain policy.)*

3. **Enable "Force logoff when logon hours expire":**
   - If using **Group Policy (GPO)**:
     - Open `gpedit.msc` → **Computer Configuration** → **Windows Settings** → **Security Settings** → **Local Policies** → **Security Options**.
     - Search for:
       **"Interactive logon: Force logoff when logon hours expire"** → **Enable**.
   - If using **Registry Editor** (for standalone machines):
     - Set:
       `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Lsa` → **LmCompatibilityLevel** = `3` (or higher).
     - Ensure logon hours are properly configured in **Active Directory Users and Computers** for each account.

4. **Verify Logon Hours in AD:**
   - Open **Active Directory Users and Computers** (`dsa.msc`).
   - For each sensitive account (e.g., admins), set **logon hours** to restrict access to business hours.
   - Example:
     - Allow logins only **Mon-Fri, 9 AM–5 PM**.

5. **Audit Logon Events:**
   - Ensure **Event ID 4624 (Successful Logon)** and **4625 (Failed Logon)** are monitored in **Event Viewer** or a SIEM (e.g., Splunk, ELK).
   - Look for anomalies (e.g., logins outside allowed hours).

6. **Test the Configuration:**
   - Log in as a test user with restricted logon hours.
   - Attempt to log in outside allowed hours → Should be **blocked with a logoff**.
