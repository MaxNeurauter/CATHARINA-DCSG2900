#### **Key Findings:**
- **LDAP server signing requirements set to "None"** is a **critical vulnerability** in Active Directory environments.
- The other policies are correctly configured for security.

---

### **3. Attack Path**
#### **Vulnerability: LDAP Server Signing Requirements (None)**
- **Attack Vector:** **LDAP Signing/Integrity Attacks (e.g., NTLM Relay, Pass-the-Hash, or Man-in-the-Middle attacks)**
- **Technical Impact:**
  - LDAP traffic is **not signed or encrypted**, allowing attackers to:
    - **Intercept and modify LDAP queries** (e.g., modifying user attributes, group memberships, or password policies).
    - **Perform NTLM relay attacks** to escalate privileges (e.g., via tools like **Responder** or **Mimikatz**).
    - **Exploit unprotected LDAP channels** to extract sensitive information (e.g., user credentials, SPNs for Kerberoasting).
  - This is a **common misconfiguration** in Active Directory environments, often exploited in **domain compromise scenarios**.

---

### **4. Remediation**
#### **Step-by-Step Fix for LDAP Server Signing Requirements:**
1. **Open Local Security Policy or Group Policy Management:**
   - Navigate to:
     **`Start → Run → secpol.msc`** (Local Security Policy)
     **OR**
     Use **Group Policy Management Console (GPMC)** (`gpmc.msc`) to apply via GPO.

2. **Navigate to the Vulnerable Setting:**
   - Go to:
     **`Local Policies → Security Options`**
     (or apply via **Domain Controller Security Policy** in GPMC).

3. **Modify LDAP Server Signing Requirements:**
   - Find the policy:
     **`Domain controller: LDAP server signing requirements`**
   - Change the setting from **"None"** to:
     - **"Sign and seal"** (most secure, encrypts and signs LDAP traffic).
     - **"Sign"** (if encryption is not required but integrity is critical).

4. **Apply and Verify:**
   - Click **Apply** → **OK**.
   - **Verify via PowerShell:**
     ```powershell
     (Get-ItemProperty 'HKLM:\SYSTEM\CurrentControlSet\Control\Lsa' -Name "LdapServerIntegrity" -ErrorAction SilentlyContinue).LdapServerIntegrity
     ```
     - Expected output for **"Sign and seal"**: `1` (or check via `reg query`).

5. **Force Group Policy Update (if using GPO):**
   ```powershell
   gpupdate /force
   ```

6. **Test LDAP Connectivity:**
   - Ensure applications (e.g., AD tools, LDAP clients) still function.
   - Use **Wireshark** or **Microsoft Message Analyzer** to confirm LDAP traffic is now signed.

7. **Document the Change:**
   - Record the modification in audit logs or a change management system.

---

### **Additional Recommendations:**
- **Enable LDAPS (LDAP over SSL/TLS):**
  - Ensure LDAP is configured to use **port 636 (LDAPS)** with a valid certificate.
  - Use **StartTLS** (port 389) if LDAPS is not feasible.
- **Restrict LDAP Access:**
  - Use **firewall rules** to limit LDAP access to trusted IPs.
  - Implement **Network Level Authentication (NLA)** for remote LDAP connections.
- **Monitor for Exploits:**
  - Use **SIEM tools** (e.g., Splunk, ELK) to detect unusual LDAP traffic.
  - Enable **Windows Event Logs** for **Event ID 5136** (LDAP sign-in attempts).

---
### **Final Note:**
This misconfiguration is **often exploited in real-world attacks** (e.g., **PrintNightmare, PetitPotam, or MITM attacks**). Remediating it **reduces the attack surface significantly** for domain compromise scenarios.
