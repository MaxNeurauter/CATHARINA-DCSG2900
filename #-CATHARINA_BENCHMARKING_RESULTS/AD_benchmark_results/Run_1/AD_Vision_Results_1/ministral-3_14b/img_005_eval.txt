### **2. Assessment**
This configuration is **Misconfigured** and introduces **significant security risks**.

#### **Key Issues:**
- **Low account lockout threshold (5 attempts)** is **too permissive** for modern security standards.
- **Short lockout duration (5 minutes)** allows brute-force attacks to continue with minimal disruption.
- **No distinction between interactive and non-interactive logons** (e.g., service accounts) increases exposure.

#### **Why?**
- Attackers can perform **credential spraying** or **brute-force attacks** efficiently without immediate account lockout.
- **Lateral movement** and **pass-the-hash attacks** become easier if legitimate users are locked out while attackers remain active.
- **Service accounts** (often non-interactive) may not trigger lockouts if misconfigured, enabling prolonged exploitation.

---

### **3. Attack Path**
#### **Potential Attack Vectors:**
1. **Brute-Force Attacks**
   - Attackers can systematically test passwords (e.g., via **Hydra, Medusa, or custom scripts**) without immediate account lockout.
   - **Impact**: Unauthorized access to user/service accounts, data exfiltration, or privilege escalation.

2. **Credential Spraying**
   - Attackers use a **small set of common passwords** across many accounts.
   - **Impact**: If even 1-2 accounts are compromised, attackers gain footholds for further attacks (e.g., **Golden Ticket, Pass-the-Hash**).

3. **Lateral Movement & Privilege Escalation**
   - If a low-privilege account is compromised, attackers may pivot to **Domain Admins** or **service accounts** (e.g., via **DCSync, Mimikatz, or Kerberoasting**).
   - **Impact**: Full domain compromise (e.g., **APT groups, ransomware deployments**).

4. **Denial-of-Service (DoS) via Account Lockout**
   - Attackers could **lock out legitimate users** (e.g., admins) by triggering false logon attempts.
   - **Impact**: Disruption of operations, inability to recover accounts without manual intervention.

#### **Technical Exploitation Example:**
- An attacker uses **Responder** or **Mimikatz** to capture hashes, then brute-forces a user account with a **5-attempt threshold**.
- If successful, they move laterally using **Pass-the-Hash (PtH)** or **Pass-the-Ticket (PtT)**.
- If failed, they **spray credentials** across other accounts before lockout kicks in.

---

### **4. Remediation – Step-by-Step Fix**
#### **Recommended Secure Configuration (Microsoft & CIS Benchmarks):**
| **Setting**                     | **Recommended Value**               | **Justification** |
|----------------------------------|-------------------------------------|-------------------|
| **Account lockout threshold**    | **10 invalid attempts (minimum)**   | Balances security and usability. |
| **Account lockout duration**     | **30+ minutes (or disabled)**       | Prevents brute-force persistence. |
| **Reset lockout counter**        | **30+ minutes**                     | Ensures failed attempts don’t reset too quickly. |
| **Sensitive accounts (Admins)**  | **0 attempts (disable lockout)**    | Prevents DoS on critical accounts. |
| **Service accounts**             | **Custom policies (e.g., 3 attempts, 15 min)** | Reduce risk while allowing some tolerance. |

#### **Implementation Steps:**
1. **Open Group Policy Management (GPMC.msc)**
   - Navigate to:
     `Computer Configuration → Windows Settings → Security Settings → Account Policies → Account Lockout Policy`

2. **Modify Settings:**
   - **Account lockout threshold** → **10 attempts** (or higher for less critical accounts).
   - **Account lockout duration** → **30 minutes** (or **disabled for admins**).
   - **Reset account lockout counter** → **30 minutes**.

3. **Apply to Domain Controllers (DCs) via GPO:**
   - Link the GPO to the **Domain Controllers OU**.
   - **Enforce** the policy (not just "Link enabled").

4. **Test the Policy:**


     ```powershell
     Get-WinEvent -FilterHashtable @{
         LogName = 'Security'
     ```

   - Provide a **break-glass procedure** for locked-out admins.
