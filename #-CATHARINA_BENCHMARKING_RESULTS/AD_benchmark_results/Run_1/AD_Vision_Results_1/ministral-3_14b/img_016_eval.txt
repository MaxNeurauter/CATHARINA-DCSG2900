### **2. Assessment**
#### **A. Secure Configurations**
- **Enforce user logon restrictions (Enabled)**
  This ensures that logon restrictions (e.g., time restrictions, workstation restrictions) are enforced, reducing lateral movement risks.
  **✅ Secure**

#### **B. Misconfigured / Critically Vulnerable Configurations**
- **Maximum lifetime for service ticket (10 hours) & user ticket (10 hours)**
  - **Issue:** The **default Kerberos ticket lifetime (10 hours for both service and user tickets)** is excessively long.
  - **Risk:** Long-lived tickets increase the attack surface for:
    - **Kerberoasting** (brute-forcing service account hashes).
    - **Golden Ticket/Silver Ticket attacks** (if an attacker obtains a TGT or service ticket).
    - **Pass-the-Ticket (PtT) attacks** (reusing valid tickets for lateral movement).
  - **Best Practice:** Microsoft and security frameworks (e.g., CIS, NIST) recommend **shorter lifetimes** (e.g., **6–8 hours for user tickets, 1–2 hours for service tickets**).
  **⚠️ Critically Vulnerable**

- **Maximum lifetime for user ticket renewal (7 days)**
  - **Issue:** A **7-day renewal window** allows attackers to reuse stolen tickets for an extended period.
  - **Best Practice:** Reduce this to **≤24 hours** to limit the window of opportunity for abuse.
  **⚠️ Vulnerable**

- **Maximum tolerance for computer clock synchronization (5 minutes)**
  - **Issue:** While **5 minutes is better than the default (5 minutes in some older AD setups)**, it is still **too lenient** for high-security environments.
  - **Risk:** Clock skew enables **Kerberos Golden Ticket attacks** (forging TGTs if time drift allows).
  - **Best Practice:** **Strictly enforce <5 minutes (ideally 0–2 minutes)** via **Group Policy** and **NTP synchronization**.
  **⚠️ Vulnerable**

### **3. Attack Path (If Exploited)**
| **Attack Vector**               | **Technical Impact**                                                                 |
|----------------------------------|--------------------------------------------------------------------------------------|
| **Kerberoasting**               | Attacker requests SPNs for service accounts, cracks hashes offline, and gains access. |
| **Golden/Silver Ticket Attacks**| If an attacker obtains a TGT (via AS-REP Roasting or Mimikatz), they can forge long-lived tickets. |
| **Pass-the-Ticket (PtT)**        | Stolen tickets (from memory or LSASS) allow lateral movement without credentials.   |
| **Time-Skew Exploitation**       | If an attacker controls a machine with slight time drift, they may forge valid Kerberos tickets. |

**Example Attack Chain:**
1. **AS-REP Roasting** → Extracts hashes of accounts with no pre-authentication.
2. **Cracking Hashes** → Obtains a service account password.
3. **Kerberoasting** → Requests a service ticket and cracks it offline.
4. **Lateral Movement** → Uses the ticket to access other systems.

### **4. Remediation (Step-by-Step Fixes)**

#### **A. Reduce Kerberos Ticket Lifetimes**
1. **Open Group Policy Management (`gpmc.msc`)**.
2. Navigate to:
   **Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Kerberos Policy**
3. **Modify the following settings:**
   - **Maximum lifetime for service ticket** → **60 minutes (1 hour)** *(or lower, e.g., 30 minutes for high-security environments)*
   - **Maximum lifetime for user ticket** → **8 hours**
   - **Maximum lifetime for user ticket renewal** → **24 hours (1 day)**
4. **Apply and enforce via Group Policy** to all domain controllers.

#### **B. Enforce Strict Time Synchronization**
1. **Ensure all domain-joined machines sync with a trusted NTP server** (e.g., `time.windows.com` or an internal NTP server).
2. **Modify Group Policy:**
   - **Computer Configuration → Policies → Administrative Templates → System → Windows Time Service → Time Providers**
   - Enable **"Configure Windows NTP Client"** and set:
     - **NTP Server:** `time.yourdomain.com` (or `0.pool.ntp.org`)
     - **Sync Interval:** `60 minutes`
3. **Verify synchronization:**
   ```powershell
   w32tm /query /status
   ```
4. **Enforce <5-minute skew** via:
   ```powershell
   # Check current clock skew (run on all DCs)
   (Get-Date).ToUniversalTime() - (Invoke-Command -ComputerName DC01 -ScriptBlock { Get-Date }).ToUniversalTime() | Select-Object Duration
   ```
   - **Best Practice:** Use **PowerShell or third-party tools** to audit and enforce time sync.

#### **C. Additional Hardening (Optional but Recommended)**
- **Disable Pre-Authentication for Service Accounts** (to prevent AS-REP Roasting):
  ```powershell
  Set-ADAccountControl -Identity "ServiceAccount" -DoesNotRequirePreAuth:$true
  ```
- **Monitor for Kerberoasting Attempts** using:
  - **Microsoft Defender for Identity**
  - **SIEM tools (Splunk, ELK) with Kerberos event logs (IDs 4769, 4776)**
- **Use Privileged Access Workstations (PAWs)** for admin tasks to limit ticket exposure.
