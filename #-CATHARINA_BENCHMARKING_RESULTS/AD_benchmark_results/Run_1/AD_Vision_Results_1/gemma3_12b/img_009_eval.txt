*   **Technical Impact:** When an account is marked as "sensitive", it triggers stricter authentication requirements. Specifically, the account's Service Principal Name (SPN) is protected with an encryption key that's only accessible to the account itself. This key is used to grant the service access to network resources. The default Kerberos encryption type is set to AES256. The sensitive flag prevents Kerberos services associated with the account from using older, weaker encryption algorithms (like DES) which are susceptible to brute-force attacks (Kerberoasting).

    If the "Account is sensitive" flag is *not* set, these service accounts are more likely to use older, easily crackable encryption types. An attacker who successfully Kerberoasts the account can obtain the password and impersonate the service, gaining unauthorized access to resources the service is authorized to access.

**4. Remediation:**

1.  **Identify Sensitive Accounts:** Determine which user accounts are or *should be* considered "sensitive." These are typically accounts that provide services (e.g., database servers, application servers, etc.).  Look for accounts that have SPNs registered to them.
2.  **Enable "Account is sensitive"**:  For each identified sensitive account, open the account's properties in Active Directory Users and Computers (ADUC).  Navigate to the "General" tab, and **check the "Account is sensitive" checkbox**.
3.  **Verify SPN registration:** Ensure that the SPNs are correctly registered for sensitive accounts. Incorrect SPNs can disrupt authentication.
4.  **Implement Kerberos Hardening (Best Practice):**
    *   Upgrade to AES256 encryption (if not already in use).  This is the default in modern Windows environments but should be verified.
    *   Consider implementing Key Version Number (KVN) rotation.
