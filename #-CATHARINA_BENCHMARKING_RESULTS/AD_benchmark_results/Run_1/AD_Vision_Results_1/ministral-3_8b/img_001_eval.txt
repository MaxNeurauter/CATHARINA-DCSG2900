### **2. Assessment**
#### **Critical Security Concern:**
- **Domain Controller LDAP signing is set to `None`** (unencrypted LDAP traffic).
  - **Classification:** **Critically Vulnerable**
    - The other settings (secure channel encryption/signing) are correctly enabled, but **LDAP signing being disabled exposes the domain to man-in-the-middle (MITM) attacks, ejecting domain controllers, and credential harvesting**.

#### **Other Settings:**
- **Domain Member & Microsoft Network Server signing policies are correctly enabled** (no immediate risk here).

---

### **3. Attack Path & Technical Impact**
#### **Primary Attack Vector: LDAP Relay / MITM Attacks**
- **LDAP Signing Disabled (`None`)**
  - Attackers can **eavesdrop, modify, or inject LDAP traffic** (e.g., Active Directory queries, user enumeration, password changes).
  - **Impact:**
    - **Kerberoasting via LDAP:** Extract user hashes (if NTLMv2 or AES reduced to dictionary attacks).
    - **AS-REP Roasting:** If Kerberos pre-auth is disabled for certain accounts, attackers can crack hashed TGTs.
    - **Domain Controller Ejection:** MITM attacks to force re-authentication and capture credentials.
    - **Pass-the-Hash/Lateral Movement:** Unencrypted LDAP allows relay attacks (e.g., Golden Ticket attacks via LDAP manipulation).

#### **Secondary Risks:**
- **No LDAP Channel Binding (LDAPS without encryption enforcement):**
  - Even if LDAPS (LDAP over SSL/TLS) is used, attackers may downgrade to unencrypted LDAP if possible.

---

### **4. Remediation: Step-by-Step Fix**
#### **Immediate Fix (Critical):**
1. **Enable LDAP Signing for Domain Controllers**
   - Open **Group Policy Management Console (GPMC)**.
   - Navigate to:
     **Computer Configuration → Policies → Windows Settings → Security Settings → Domain Controller Policy → LDAP Client Signing Requirements**.
   - Change the setting from **None** to **Signing Required**.
   - Apply and force a GP update (`gpupdate /force` on DCs).

2. **Enforce LDAPS (LDAP over SSL/TLS)**
   - Ensure all LDAP traffic is forced to use **LDAPS (port 636)**.
   - Configure **Certificate Services** to issue certificates for LDAP (or use a trusted CA).
   - Set **Channel Binding** in GPO:
     **Computer Configuration → Policies → Administrative Templates → System → Kerberos → "Require Channel Binding for LDAP Signing"** → **Enabled**.

3. **Audit & Verify**
   - Check Event Logs for LDAP errors (Event ID 5825, 5826).
   - Use **WireShark** or **Process Monitor** to confirm LDAP traffic is signed.

#### **Additional Hardening (Recommended):**
- **Disable LDAP over plaintext (port 389)** at the firewall level.
- **Enable LDAP Integrity (if supported)** for additional protection.
- **Monitor for LDAP relay attacks** using tools like **BloodHound** or **ADAudit Plus**.
