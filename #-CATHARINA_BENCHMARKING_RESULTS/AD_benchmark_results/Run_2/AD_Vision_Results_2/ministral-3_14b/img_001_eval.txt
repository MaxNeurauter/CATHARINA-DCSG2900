### **3. Attack Path (If Vulnerable)**
#### **Vulnerability: LDAP Signing Disabled (Domain Controller)**
- **Attack Vector:** **LDAP Signing/Integrity Attacks (e.g., NTLM Relay, Forged LDAP Responses, Pass-the-Hash)**
- **Technical Impact:**
  - An attacker on the network can **forge LDAP responses** (e.g., spoofing authentication replies).
  - **NTLM Relay Attacks** become possible if LDAP is used for authentication (e.g., legacy apps or misconfigured services).
  - **Kerberos authentication bypass** if LDAP is used for password changes or attribute modifications.
  - **Information Disclosure:** Attackers can intercept and modify LDAP traffic without detection.

#### **Example Attack Scenario:**
1. An attacker captures LDAP traffic (e.g., via ARP spoofing or man-in-the-middle).
2. Since signing is disabled, they can **modify responses** (e.g., changing a user’s password or group membership).
3. If NTLM is used for authentication, they can **relay hashes** to gain access to other systems.

---

### **4. Remediation (Step-by-Step Fix)**
#### **For LDAP Signing (Domain Controller)**
1. **Open Group Policy Management Console (GPMC.msc)** or **Local Security Policy (secpol.msc)**.
2. Navigate to:
   **Computer Configuration → Windows Settings → Security Settings → Local Policies → Security Options**
3. Locate:
   - **"Domain controller: LDAP server signing requirements"**
4. **Change the setting to:**
   - **"Require signing"** (or **"Require signing and sealing"** for stricter security).
5. **Apply and enforce the policy** via:
   - **Domain-wide GPO** (if managing multiple DCs) or **local policy** (for a single DC).
6. **Verify enforcement** using:
   ```powershell
   Get-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" -Name "LmCompatibilityLevel"
   ```
   - Ensure it reflects the new signing requirements.

#### **Additional Hardening (Recommended)**
- **Enable LDAP Channel Binding** (if using Windows Server 2012 R2+):
  - Requires **LDAP over TLS (LDAPS)** or **StartTLS** for encrypted sessions.
- **Disable NTLM for LDAP** (if possible) and enforce **Kerberos-only authentication**.
- **Audit LDAP access** via **Windows Event Logs** (Event ID 5136 for LDAP signing failures).

#### **For SMB & Secure Channel (Already Secure)**
- No changes needed, but ensure:
  - **SMBv1 is disabled** (use SMBv2/v3 with signing).
  - **Firewall rules** restrict LDAP/SMB to trusted hosts only.
