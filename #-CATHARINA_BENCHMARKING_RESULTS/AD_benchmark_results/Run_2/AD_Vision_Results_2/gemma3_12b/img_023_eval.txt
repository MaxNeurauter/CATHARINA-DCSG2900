**3. Attack Path & Technical Impact:**

The "Account is sensitive" attribute in Active Directory is critical for security, especially when Kerberos authentication is in use. When this flag is *unchecked*, it exposes a vulnerability to **Kerberoasting**.

*   **Kerberoasting Explanation:** Kerberoasting is an attack where an attacker obtains a Kerberos ticket for a service principal name (SPN). These tickets contain the service's password hash. If the "Account is sensitive" flag is *not* set, the Kerberos ticket is encrypted with a weaker algorithm (DES). Attackers can relatively easily crack DES hashes using readily available tools. This gives them the service account’s password, allowing them to impersonate the service, escalate privileges, or access sensitive data.

**4. Remediation:**

Here's a step-by-step remediation:

1.  **Identify Affected Accounts:** Determine which accounts in your domain have SPNs registered and for which the "Account is sensitive" flag is currently unchecked. You can use PowerShell commands to achieve this.  Example:
    ```powershell
    Get-ADComputer -Filter "UserAccountControl -like '*26*'" -Properties DistinguishedName,UserAccountControl | Select-Object Name,DistinguishedName
    ```
2.  **Enable "Account is sensitive":**  For *each* affected account, enable the "Account is sensitive" flag. There are several methods:
    *   **Active Directory Users and Computers (ADUC):**  Open ADUC, find the account, go to the "Account" tab, and check the box "Account is sensitive".
    *   **PowerShell:** Use the `Set-ADAccount` cmdlet:
        ```powershell
        Set-ADAccount -Identity "username" -AccountSensitive $true
        ```  Replace `"username"` with the actual username.
3.  **Verify Encryption Type:** After setting the flag, verify that the Kerberos tickets for the affected service accounts are now encrypted using AES (Advanced Encryption Standard) instead of DES. This can be done using tools like `klist` on a domain-joined machine or by inspecting the Kerberos ticket using Wireshark.
4.  **Review Service Principal Names (SPNs):** Review all SPNs registered for accounts and services to ensure they are correct and only assigned to legitimate accounts. Remove any unused or orphaned SPNs.
