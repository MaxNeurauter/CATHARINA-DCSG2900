### **3. Attack Paths & Technical Impact**
#### **A. Kerberoasting / AS-REP Roasting**
- **Vector:** If `Authenticated Users` can enable **reversible encryption** on service accounts, attackers can **Kerberoast** (extract SPN hashes) or **AS-REP Roast** (if `Do Not Require Kerberos Pre-Auth` is enabled).
- **Impact:** Compromised service account credentials can be used for **lateral movement** or **privilege escalation**.

#### **B. Anonymous Enumeration (Reconnaissance)**
- **Vector:** `ANONYMOUS LOGON` has `Read` access to descendant objects.
- **Impact:** Attackers can **enumerate users, groups, and computers** without authentication, aiding in **phishing campaigns** or **targeted attacks**.

#### **C. Privilege Escalation via Token Impersonation**
- **Vector:** `Pre-Windows 2000 Compatible Access` + `SYSTEM Full Control` could allow an attacker to **impersonate tokens** and escalate privileges.
- **Impact:** Gain **Domain Admin** or **Enterprise Admin** rights.

#### **D. Password Expiration Bypass**
- **Vector:** `Authenticated Users` can unset password expiration.
- **Impact:** Attackers could **disable password rotation** for accounts, making them harder to detect if compromised.

### **4. Remediation: Step-by-Step Fix**
#### **Step 1: Remove Anonymous Logon Read Access**
1. Open **Active Directory Users and Computers**.
2. Navigate to the **OU/GPO** with the misconfigured permissions.
3. Right-click → **Properties** → **Security** tab.
4. Remove the **`ANONYMOUS LOGON`** entry with **Read** permissions.
5. Click **Apply** → **OK**.

#### **Step 2: Restrict `Authenticated Users` Permissions**
1. In the **Security** tab, locate the **`Authenticated Users`** entry.
2. **Remove** the following permissions:
   - `Enable per user reversibly encrypted password`
   - `Unset password not required bit`
3. Replace with **only necessary permissions** (e.g., `Read` if required).
4. Click **Apply** → **OK**.

#### **Step 3: Restrict `SYSTEM` Permissions**
1. In the **Security** tab, locate the **`SYSTEM`** entry.
2. **Remove** `Full Control` if not explicitly required.
3. Grant only **necessary permissions** (e.g., `Read` or `Replicating Directory Changes`).
4. Click **Apply** → **OK**.

#### **Step 4: Remove Pre-Windows 2000 Compatible Access**
1. In the **Security** tab, locate entries with **`Pre-Windows 2000 Compatible Access`**.
2. **Remove** these entries unless explicitly required for legacy compatibility.
3. Click **Apply** → **OK**.

#### **Step 5: Review `Domain Admins` and `Enterprise Admins` Permissions**
1. Ensure these groups **only have necessary permissions** (e.g., `Full Control` should be limited to **specific OUs**, not all descendant objects).
2. If `Full Control` is granted to **all descendant objects**, **narrow it down** to only critical OUs (e.g., **GPO containers**).

#### **Step 6: Enforce Least Privilege for Service Accounts**
1. Ensure **service accounts** (e.g., `svc_*` accounts) **do not have reversible encryption enabled**.
2. Set **password expiration** and **complex passwords** for all accounts.
3. Disable **`Do Not Require Kerberos Pre-Authentication`** for service accounts.

#### **Step 7: Audit Changes with Security Logging**
1. Enable **Advanced Audit Policy Configuration** for:
   - **Directory Service Access** (e.g., `Object Access` auditing).
   - **Account Management** (e.g., `Password Change` auditing).
2. Monitor for **unusual permission changes** using **Security Event ID 4662** (Special Permission Modification).

#### **Step 8: Apply Security Best Practices**
- **Principle of Least Privilege (PoLP):** Ensure only necessary groups/users have permissions.
- **Separation of Duties:** Avoid granting `Full Control` to broad groups like `Domain Admins` on GPOs.
- **Regular Audits:** Use **Group Policy Permissions Analyzer** or **PowerShell** (`Get-Acl`) to check permissions.
