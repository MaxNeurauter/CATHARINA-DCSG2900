### 2. Assessment  
**Misconfigured**. A standard user account should *never* have a `servicePrincipalName` (SPN) set. SPNs are reserved for **service accounts** (e.g., computer accounts or dedicated service accounts) to enable Kerberos authentication for services. This misconfiguration exposes the account to a specific attack vector.  

### 3. Attack Path  
- **Attack Vector**: **Kerberoasting**.  
- **Technical Impact**:  
  - An attacker can request a Kerberos TGS (Ticket-Granting Service) ticket for the SPN (e.g., `CIFS/thewall.north.sevenkingdoms.local`).  
  - The TGS ticket is encrypted with the *user account’s password hash* (not the service account’s, since this is a user account). If the password is weak or reused, the attacker can crack the ticket to obtain the user’s password.  
  - This compromises the `jon.snow` account, potentially enabling lateral movement, data exfiltration, or privilege escalation.  

### 4. Remediation  
**Step-by-Step Fix**:  
1. **Identify all user accounts with SPNs**:  
   Run in PowerShell:  
   ```powershell
   Get-ADUser -Filter "servicePrincipalName -like '*' -and objectClass -eq 'user'" -Properties servicePrincipalName | Select-Object Name, servicePrincipalName
   ```  
2. **Remove the SPN from the user account**:  
   Use PowerShell to delete the SPN:  
   ```powershell
   Set-ADUser -Identity "jon.snow" -Remove @{servicePrincipalName="CIFS/thewall.north.sevenkingdoms.local; HT"}
   ```

4. **Enforce policy**:
