- **Topology Type**: **Hierarchical (2-tier)**. The LAN and DMZ are distinct access-layer segments, with the router acting as the distribution/core layer. Traffic flows through the router for inter-segment and Internet access.

#### 2. Purpose & Architecture
The design is a **basic segmented corporate network with a DMZ**. The LAN (private internal network) hosts user endpoints, while the DMZ (public-facing segment) isolates servers exposed to the Internet. The 2901 router serves as the sole gateway for all traffic between the LAN, DMZ, and Internet, with no explicit firewall between segments.

#### 3. Strengths (Pros)
- **Clear Segmentation**: The DMZ isolates public-facing servers from the internal LAN, reducing the risk of external attacks spreading to critical assets.
- **Logical Grouping**: PCs are grouped in the LAN (internal use), servers in the DMZ (public access), enabling targeted management and security policies.
- **Simplicity**: The 2-tier design is easy to implement and troubleshoot for small-scale environments.

#### 4. Weaknesses & Risks (Cons)
- **Critical Single Points of Failure (SPOFs)**:
  - **2901 Router1**: Failure halts all Internet access and inter-segment traffic (LAN ↔ DMZ). No backup router is present.
  - **2960-24TT Switches**: Each switch is a SPOF for its segment (e.g., LAN switch failure isolates all PCs; DMZ switch failure isolates servers).
- **Inadequate Security Boundaries**:
  - **No Dedicated Firewall**: The router (2901) lacks advanced security features (e.g., stateful inspection, application-layer filtering). Relying on basic ACLs is insufficient for modern threats.
  - **Weak DMZ Isolation**: The router *only* separates the DMZ from the Internet/LAN but does not enforce strict unidirectional traffic (e.g., no firewall between DMZ and LAN to block server-to-LAN attacks).
- **Lack of Redundancy & Scalability**: The design has no redundancy (e.g., no HSRP/VRRP for the router, no stacked switches). Growth will quickly cause congestion and reliability issues.
- **No VLAN Segmentation**: All PCs in the LAN and all servers in the DMZ share the same broadcast domain, increasing broadcast traffic and lateral attack risks.

#### 5. Recommendations
1. **Deploy a Firewall and Redundant Router Pair**:
   - Add a **dedicated firewall (e.g., Cisco ASA)** between the Internet and Router1 to enforce stateful inspection. Place a second firewall between Router1 and the DMZ to isolate the DMZ from the LAN. Complement with **HSRP/VRRP on two 2901 routers** to eliminate the SPOF at the edge. *Rationale: This aligns with Cisco's best practices for secure DMZs (2-firewall model) and achieves high availability (HA).*

2. **Implement VLANs and Redundant Switching**:
   - **Segment LAN/DMZ with VLANs**: Create VLANs for PCs (e.g., VLAN 10) and servers (e.g., VLAN 20) on both 2960-24TT switches. Configure the router for inter-VLAN routing via sub-interfaces. *Rationale: Reduces broadcast domains, improves security, and enables granular policies (e.g., block server traffic from the LAN).*
   - **Add Switch Redundancy**: Install a second switch in the LAN/DMZ, link them with a trunk, and enable **STP (802.1w)** to prevent loops. *Rationale: Eliminates SPOFs in the access layer and scales for growth.*

3. **Strengthen DMZ Isolation**:
   - **Enforce Strict DMZ Policies**: On the router (or new firewall), deploy **unidirectional ACLs** (e.g., block all LAN → DMZ traffic; allow only DMZ → LAN for specific ports like NTP). *Rationale: Limits server attacks from the DMZ to the internal network, adhering to the principle of least privilege.*
